Projet CRM

Bienvenue sur le projet CRM. J'ai pu participer en tant que stagiaire chez Norsys. Ce projet a été développé par l'équipe de développement de l'entreprise. En tant que stagiaire chez Norsys, j'ai contribué à ce projet. Il a commencé en février 2023 et devrait se terminer en juillet 2023. Les clients de ce projet sont les commerciaux de l'entreprise Norsys.

L'objectif principal du projet est de créer une interface rapide et intuitive pour les commerciaux, où chacun dispose d'un espace numérique personnalisé. Cet espace contient la liste des comptes clients de l'entreprise. Lorsqu'un commercial sélectionne un compte client, il peut accéder à une multitude d'informations, telles que les salariés travaillant avec ce client (RH, développeurs, chefs de projet, etc.), les contacts associés, les services liés, et plus encore. Les commerciaux ont également la possibilité de passer des appels en visioconférence directement depuis l'interface. Cela leur permet de communiquer plus facilement avec les clients et de renforcer les interactions.
L'interface CRM du projet vise à faciliter le travail des commerciaux en leur fournissant toutes les informations nécessaires sur les comptes clients de manière claire et organisée. Cela leur permet de mieux comprendre les relations entre l'entreprise Norsys et ses clients, ainsi que de gérer efficacement leurs interactions avec eux.

Le sprint

Le projet fonctionne par sprint. Mais alors qu'est-ce que c'est ? Un sprint est une période de temps définie dans la méthodologie de développement Agile. Pendant un sprint, l'équipe de développement se concentre sur un ensemble spécifique de fonctionnalités ou de tâches à accomplir. Habituellement, un sprint dure entre une à quatre semaines. À la fin de chaque sprint, l'équipe de développement présente les fonctionnalités développées à Julien VIERNO pour validation. Julien VIERNO est un commercial de l'entreprise qui est le client référent de ce projet. Cette approche itérative par sprint permet d'obtenir des retours réguliers du client tout au long du processus de développement et d'ajuster les fonctionnalités en fonction de ses besoins et de ses attentes.

    Tests unitaires

    Les tests unitaires sont une pratique de développement logiciel qui consiste à écrire des tests automatisés pour vérifier le bon fonctionnement des unités individuelles de code, telles que des classes, des méthodes ou des fonctions, dans une application.

    Dans le contexte d'un projet Java Angular, les tests unitaires sont utilisés pour vérifier le comportement attendu des différentes parties de votre code Angular, y compris les composants, les services et les classes utilitaires. Ils vous aident à détecter les erreurs ou les régressions potentielles plus tôt dans le processus de développement, ce qui facilite la détection et la résolution des problèmes.

    Les tests unitaires permettent de valider le fonctionnement isolé des unités de code. Ils vérifient si chaque unité de code répond correctement aux différentes entrées et conditions attendues, en s'assurant que le code produit les résultats escomptés.

    Voici quelques raisons pour lesquelles les tests unitaires sont importants :

    • Détection précoce des erreurs
    • Facilité de maintenance
    • Régression minimale
    • Confiance dans le code

    Hibernate

    Hibernate est un framework de persistance objet pour Java qui facilite le stockage et la récupération des objets Java dans une base de données relationnelle.

    Lexique

    Peer-Programming

    La peer programming (pair programming en francais), ou programmation en binôme, est une méthode de développement logiciel où deux programmeurs travaillent ensemble sur le même ordinateur, en partageant le clavier et l'écran, pour collaborer étroitement dans la résolution de problèmes, la conception et la rédaction de code.

    Collaps

    Collaps est un élément ou un composant d'interface utilisateur qui permet de réduire ou d'expandre du contenu de manière interactive, généralement à l'aide d'un bouton ou d'un lien, pour économiser de l'espace à l'écran.

    Mob prog

    Le "mob programming" est le terme utilisé pour décrire la pratique de plusieurs programmeurs travaillant ensemble sur un même projet ou un même ensemble de fonctionnalités.

    Module

    En Angular, le terme Module désigne un ensemble de fonctionnalités d'une application, il donne essentiellement des informations angulaires sur les fonctionnalités de celle-ci

    Framework

    Un framework est un ensemble d'outils, de bibliothèques et de conventions de développement préétablies qui facilitent la création et la gestion de logiciels. Il fournit une structure de base pour le développement d'applications en regroupant des fonctionnalités communes et en fournissant des abstractions pour simplifier les tâches récurrentes. Les frameworks permettent aux développeurs de gagner du temps en réutilisant du code existant et en suivant des bonnes pratiques établies. Ils offrent également une architecture cohérente et favorisent une approche modulaire et évolutive du développement logiciel. En résumé, un framework est un ensemble d'outils et de conventions qui accélèrent le processus de développement en fournissant une structure et des fonctionnalités prêtes à l'emploi.

    M R

    Les merge requests (MR), également appelées pull requests, sont une fonctionnalité essentielle dans les systèmes de contrôle de version tels que Git. Elles permettent aux développeurs de proposer des modifications de code à une branche principale ou à un référentiel central. Les merge requests permettent une collaboration efficace en facilitant la révision et la validation des changements par les pairs. Elles offrent également un historique clair des modifications apportées au code et permettent de résoudre les conflits potentiels lors de la fusion des branches. Les merge requests sont donc un outil précieux pour assurer la qualité du code et la coordination dans les projets de développement logiciel.

    Refactor

    Le refactoring est le processus d'amélioration du code existant sans en modifier le comportement externe. Il vise à rendre le code plus lisible, maintenable et performant. Cela implique des actions telles que l'élimination des duplications, la simplification des algorithmes et l'application de bonnes pratiques de codage. Le refactoring est une pratique essentielle pour maintenir un code de qualité et faciliter son évolution.

    T D D

    Le TDD (Test-Driven Development), ou développement piloté par les tests, est une approche de développement logiciel dans laquelle les tests unitaires sont écrits avant même d'écrire le code de production. Le processus TDD suit généralement trois étapes : écrire un test unitaire qui échoue, écrire le code de production pour faire passer le test et enfin refactoriser le code pour l'améliorer tout en conservant les tests fonctionnels.

    Mock

    Le mocking consiste à créer des objets factices qui imitent le comportement des objets réels dans le système. Ces objets fictifs, appelés "mocks", permettent de simuler les interactions entre les différentes parties du système et de vérifier les appels aux méthodes et les retours de valeur.

    Stubs

    Le stubbing, quant à lui, est une technique similaire mais se concentre sur le remplacement de certaines parties du système par des objets factices appelés "stubs". Les stubs sont utilisés pour fournir des réponses prédéfinies lorsqu'une méthode est appelée, sans simuler complètement le comportement du véritable objet.
